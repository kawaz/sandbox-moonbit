# ドキュメンテーションルール

## docs/ の構成

```
docs/
  *.md              # 最新のドキュメント（フラットに配置）
  archives/         # 過去の設計記録（DR）のアーカイブ
    NNN-title.md    # 番号プレフィクスで時系列管理
```

- `docs/` 直下は**常に最新**。古い DR が混在しないよう維持する
- ワークスペース固有の資料はそのワークスペース内に配置（default に集約しない）

## 設計記録（DR: Design Record）

技術判断・設計決定を行った際に記録する。

### 記録内容

- 決定事項と理由（なぜその選択か、不採用案とその理由も）
- 調査結果・PoC の結果
- **生のチャット履歴を付録として残す**（無機質なまとめだけでは思想・コンテキストが失われる）

### ライフサイクル

1. 活発な設計議論中 → `docs/` 直下に配置
2. 設計が確定・次フェーズへ移行 → `docs/archives/NNN-title.md` へ移動
3. 番号は連番（`001-`, `002-`, ...）。歴史順を把握可能にする

## 知見の逐次ドキュメント化

技術調査・PoC で得た知見は即座に記録する。

対象:
- 技術的な発見（動いた/動かなかった理由）
- ハマりポイントと回避策
- 外部プロジェクトの調査結果

記録先:
- ワークスペース内 `findings.md` — そのワークスペースで得た知見
- `docs/*.md` — リポジトリ横断で再利用する知見
